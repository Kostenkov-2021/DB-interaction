# Групповая работа 1. HA-кластер ClickHouse 2x2

Костенков Данил Денисович, P4150

## Что сделано

В каталоге собран воспроизводимый стенд отказоустойчивого ClickHouse-кластера:

- кластер `production`: 2 шарда x 2 реплики;
- 3 узла ClickHouse Keeper для кворума;
- Nginx как HTTP-балансировщик ClickHouse API на порту `8123`;
- Prometheus для сбора `/metrics` со всех ClickHouse-узлов;
- Grafana с автоматическим provisioning datasource и dashboard;
- таблицы `metrics_local` на `ReplicatedMergeTree` и `metrics_distributed` на `Distributed`;
- генератор тестовой телеметрии на 5 000 000 строк;
- автоматизированные проверки и протоколы fault injection в `checks/`.

Конфигурация ClickHouse шаблонизирована через `docker-compose.yml`: все четыре сервера используют один набор общих volume и один файл `config/clickhouse/macros.xml`, а значения `{shard}` и `{replica}` берутся из переменных окружения `CLICKHOUSE_SHARD` и `CLICKHOUSE_REPLICA`. Общие параметры сервисов вынесены в YAML anchors `x-clickhouse-common` и `x-keeper-common`.

## Описание проекта

Проект моделирует production-подобную инфраструктуру для хранения и анализа телеметрических метрик. В качестве основной базы данных используется ClickHouse, потому что он хорошо подходит для аналитических нагрузок: быстро выполняет агрегации по большим объемам данных, эффективно сжимает колоночные данные и поддерживает горизонтальное масштабирование через шардирование.

Стенд построен вокруг сценария сбора метрик от приложений. Каждая запись содержит время измерения, логический хост, имя метрики и числовое значение. Такая модель близка к реальным системам мониторинга: данные пишутся большим потоком, затем анализируются агрегатами по времени, хостам и типам метрик.

Основная цель проекта - показать не только работу одной базы данных, а полный контур эксплуатации:

- хранение данных в распределенном ClickHouse-кластере;
- репликация данных внутри каждого шарда;
- балансировка клиентских HTTP-запросов;
- сбор инфраструктурных метрик;
- визуализация состояния кластера;
- автоматизированные проверки;
- документированная проверка отказоустойчивости.

## Используемые технологии

### Docker Compose

`docker-compose.yml` описывает всю инфраструктуру как код. Одной командой поднимаются 10 сервисов: 4 ClickHouse, 3 ClickHouse Keeper, Nginx, Prometheus и Grafana. Такой подход делает стенд воспроизводимым: другой человек может поднять тот же набор сервисов без ручной настройки серверов.

В Compose используются шаблоны через YAML anchors:

- `x-clickhouse-common` - общие настройки ClickHouse-сервисов;
- `x-keeper-common` - общие настройки Keeper-сервисов.

Это снижает дублирование и делает конфигурацию проще для сопровождения. Различия между ClickHouse-узлами задаются через `hostname`, `container_name`, volume с данными и переменные окружения `CLICKHOUSE_SHARD`, `CLICKHOUSE_REPLICA`.

### ClickHouse

ClickHouse используется как распределенная аналитическая СУБД. В проекте развернут кластер `production` с топологией 2 шарда x 2 реплики:

- shard 1: `ch-s1-r1`, `ch-s1-r2`;
- shard 2: `ch-s2-r1`, `ch-s2-r2`.

Шардирование распределяет данные между двумя независимыми группами узлов. Это позволяет масштабировать хранение и обработку запросов горизонтально. Репликация внутри каждого шарда защищает от потери одного узла: если одна реплика недоступна, данные остаются доступными на второй.

Для хранения используется таблица `metrics_local`:

```sql
ENGINE = ReplicatedMergeTree(
    '/clickhouse/tables/{shard}/metrics_local',
    '{replica}'
)
```

`ReplicatedMergeTree` выбран потому, что это основной движок ClickHouse для надежного хранения с репликацией. Он хранит данные локально на узле, а согласование реплик выполняет через ClickHouse Keeper.

Поверх локальных таблиц создана таблица `metrics_distributed`:

```sql
ENGINE = Distributed(
    'production',
    'default',
    'metrics_local',
    xxHash64(host, metric_name)
)
```

`Distributed` не хранит данные самостоятельно, а маршрутизирует запросы и вставки в локальные таблицы кластера. Благодаря этому клиент может обращаться к одной логической таблице, а ClickHouse сам распределяет работу по шардам.

### ClickHouse Keeper

ClickHouse Keeper используется как координационный сервис для репликации. Он хранит служебное состояние реплицируемых таблиц: список реплик, очереди репликации, лидерство, состояние частей данных.

В проекте подняты 3 Keeper-узла:

- `keeper-1`;
- `keeper-2`;
- `keeper-3`.

Три узла нужны для кворума. При потере одного Keeper кворум сохраняется, поэтому вставки в `ReplicatedMergeTree` продолжают работать. При потере двух Keeper кворум теряется, и реплицируемая таблица переходит в readonly-режим для операций записи. Этот сценарий проверяется в `scripts/fault_injection.sh`.

### Nginx

Nginx используется как HTTP-балансировщик для ClickHouse API. Он слушает порт `18123` на хосте и проксирует запросы на порт `8123` всех четырех ClickHouse-узлов.

Балансировка настроена round-robin по четырем upstream:

```nginx
server ch-s1-r1:8123 max_fails=2 fail_timeout=10s;
server ch-s1-r2:8123 max_fails=2 fail_timeout=10s;
server ch-s2-r1:8123 max_fails=2 fail_timeout=10s;
server ch-s2-r2:8123 max_fails=2 fail_timeout=10s;
```

Выбран вариант балансировки по всем 4 узлам, потому что `metrics_distributed` доступна на каждом ClickHouse-сервере. Любой узел может принять HTTP-запрос, стать координатором distributed-запроса и собрать результат с нужных шардов.

Passive health checks реализованы стандартными средствами Nginx: если узел несколько раз не отвечает, он временно исключается из upstream. Access-log настроен в JSON-формате, чтобы логи было удобно читать и обрабатывать автоматическими инструментами.

### Prometheus

Prometheus собирает технические метрики ClickHouse через встроенный endpoint `/metrics` на порту `9363`. В конфигурации `config/clickhouse/cluster.xml` включены:

- `metrics`;
- `events`;
- `asynchronous_metrics`;
- `status_info`.

В `config/prometheus/prometheus.yml` перечислены все четыре ClickHouse-узла. Prometheus нужен для независимого наблюдения за состоянием кластера: количеством запросов, памятью, активностью MergeTree и репликацией.

### Grafana

Grafana используется для визуализации метрик из Prometheus. В проекте настроен автоматический provisioning:

- `monitoring/provisioning/datasources.yml` добавляет Prometheus datasource;
- `monitoring/provisioning/dashboards.yml` подключает каталог dashboard;
- `monitoring/dashboards/clickhouse.json` хранит экспортированный dashboard.

Dashboard содержит панели для ключевых показателей:

- количество строк в MergeTree-частях;
- количество запросов в секунду;
- активность репликации;
- использование памяти.

Такой подход важен для воспроизводимости: dashboard не создается вручную в UI, а хранится в репозитории как JSON.

### Makefile

Makefile дает единый интерфейс для работы со стендом. Вместо длинных команд Docker Compose используются короткие цели:

- `make up`;
- `make init`;
- `make load`;
- `make test`;
- `make fault`;
- `make down`.

Это снижает вероятность ошибки при запуске и делает README проще: пользователь выполняет фиксированную последовательность команд.

### Shell-скрипты

В проекте используются shell-скрипты для автоматизации:

- `generate_data.sh` генерирует тестовые данные внутри ClickHouse через `numbers(N)`;
- `run_checks.sh` сохраняет результаты проверок в текстовые файлы;
- `fault_injection.sh` останавливает и запускает контейнеры для проверки отказоустойчивости.

Генерация данных выполняется SQL-запросом внутри ClickHouse, а не внешним построчным генератором. Это быстрее и проще: ClickHouse сам создает миллионы строк через системную табличную функцию `numbers`.

## Подход к данным

Тестовая таблица описывает поток телеметрии:

| Поле | Тип | Назначение |
|---|---|---|
| `timestamp` | `DateTime` | время измерения |
| `host` | `LowCardinality(String)` | логический источник метрики |
| `metric_name` | `LowCardinality(String)` | имя метрики |
| `value` | `Float64` | числовое значение |

`LowCardinality(String)` выбран для `host` и `metric_name`, потому что эти поля содержат повторяющиеся строковые значения. ClickHouse хранит такие значения через словарь, что уменьшает объем данных и ускоряет группировки.

Партиционирование выполнено по месяцу:

```sql
PARTITION BY toYYYYMM(timestamp)
```

Это стандартный подход для временных данных: старые партиции можно обслуживать, переносить или удалять независимо.

Сортировка:

```sql
ORDER BY (host, metric_name, timestamp)
```

Такой порядок удобен для запросов по конкретному хосту, метрике и временному диапазону. Данные одного логического ряда лежат близко друг к другу, что помогает ClickHouse читать меньше лишних гранул.

Ключ шардирования:

```sql
xxHash64(host, metric_name)
```

Он детерминированно распределяет разные пары `host + metric_name` по шардам. Это даёт почти равномерный баланс и сохраняет один логический ряд на одном шарде.

## Подход к отказоустойчивости

Отказоустойчивость проверяется не только декларативно, а через сценарии в `scripts/fault_injection.sh`.

Проверяются четыре ситуации:

| Сценарий | Ожидаемое поведение |
|---|---|
| Потеря одной реплики ClickHouse | запросы через Nginx продолжают работать |
| Потеря всего шарда | distributed-запрос получает ошибку по недоступному shard |
| Потеря одного Keeper | кворум сохраняется, INSERT работает |
| Потеря кворума Keeper | INSERT невозможен, SELECT по существующим данным работает |

Результаты записываются в `checks/fault_scenarios.txt`.

## Подход к проверкам

Проверки сохраняются в каталог `checks/`, потому что по стратегии проекта результаты должны быть текстовыми файлами. Основной скрипт `run_checks.sh` проверяет:

- состав кластера `production`;
- количество строк на каждой реплике;
- баланс строк по шардам;
- состояние `system.replicas`;
- корректность запросов через `metrics_distributed`;
- доступность ClickHouse через Nginx.

После полного запуска в стенде загружено больше 5 000 000 строк. В текущем протоколе после fault injection счетчик равен `5 000 001`, потому что сценарий с потерей одного Keeper специально выполняет один успешный INSERT для доказательства сохранения кворума.

## Проверки

После `make test` в каталоге `checks/` появляются:

- `cluster_status.txt` - состав кластера `production`;
- `data_distribution.txt` - распределение строк по шардам и репликам;
- `replication_status.txt` - состояние `system.replicas`;
- `distributed_queries.txt` - контрольные запросы через `metrics_distributed`;
- `nginx_failover.txt` - проверка доступности через Nginx при остановке одной реплики.

`make fault` дополнительно пишет `checks/fault_scenarios.txt` с четырьмя сценариями:

1. потеря реплики `ch-s1-r2`;
2. потеря шарда `ch-s2-r1` и `ch-s2-r2`;
3. потеря одного Keeper `keeper-1`;
4. потеря кворума Keeper `keeper-1` и `keeper-2`.

Скрипт после каждого сценария возвращает остановленные контейнеры обратно.

## Архитектура

Nginx балансирует HTTP-запросы round-robin по всем четырем ClickHouse-узлам. Выбран вариант с 4 upstream, потому что таблица `metrics_distributed` существует на каждом сервере, а ClickHouse сам координирует распределенный запрос по шардам. Passive health checks настроены параметрами `max_fails=2 fail_timeout=10s`: временно недоступный узел исключается из балансировки без отдельного active-check модуля.

Вставка идёт в `metrics_distributed` с ключом шардирования `xxHash64(host, metric_name)`. Для потока телеметрии это даёт стабильное и равномерное распределение по шардам: один и тот же логический ряд попадает на один shard, а разные хосты и метрики хорошо перемешиваются.

Prometheus собирает метрики с портов `9363` внутри Docker-сети. Grafana получает Prometheus datasource и dashboard автоматически через `monitoring/provisioning`.
