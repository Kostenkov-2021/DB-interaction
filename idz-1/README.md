## Что сделано

Реализована база данных интернет-магазина в PostgreSQL:

- `orders_raw` — ненормализованная плоская таблица, имитирующая выгрузку из Excel, с 1200 заказами.
- Последовательные миграции `UNF -> 1NF -> 2NF -> 3NF`.
- Итоговая 3NF-схема: `customers`, `addresses`, `categories`, `products`, `orders`, `order_items`.
- OLTP-запросы, B-tree/partial/GIN-индексы, материализованное представление и денормализация в таблицу.
- ER-диаграмма в PlantUML: `schema.puml`.
- Текстовые результаты проверок в каталоге `checks/`.

## Аномалии в UNF

Аномалия вставки: новый товар нельзя сохранить в каталоге до появления заказа с этим товаром. Например, если магазин хочет добавить товар `Gaming Chair` до первой продажи, в `orders_raw` для него нет отдельного места: сведения о товарах хранятся только внутри строк заказа в составных полях `product_names` и `product_prices`.

Аномалия обновления: контактные данные клиента дублируются в разных заказах. `customer001@shop.test` встречается в нескольких строках; если изменить телефон только в одной строке, в `orders_raw` появятся противоречивые телефоны одного клиента.

Аномалия удаления: если удалить единственный заказ, где встречался редкий товар, вместе с заказом потеряются сохранённые сведения о названии и цене этого товара, которые присутствуют в единственном экземпляре.

## Нормальные формы

1NF устраняет повторяющиеся группы: составные поля `product_names`, `product_prices`, `product_quantities` разбиваются на отдельные строки в `orders_1nf`. Ключ строки заказа становится составным: `(order_id, line_no)`.

2NF устраняет частичные зависимости от составного ключа. Данные клиента зависят от заказа, данные товара зависят от товара, а количество и цена на момент заказа относятся к конкретной строке заказа. Эти зависимости вынесены в `customers_2nf`, `products_2nf`, `orders_2nf`, `order_items_2nf`.

3NF устраняет транзитивные зависимости. Категория товара вынесена в `categories`, повторяющиеся адреса доставки — в `addresses`. Итоговый заказ ссылается на клиента и выбранный адрес.

BCNF и 4NF: итоговая схема осознанно проектируется до 3NF. Для выбранной предметной области BCNF фактически выполняется за счёт объявленных ключей и уникальных ограничений. 4NF не даст полезного дополнительного разбиения в минимальной схеме, потому что независимые многозначные факты, например несколько тегов товара или несколько каналов связи клиента, не получится смоделироватья; при добавлении их придётся выносить в отдельные связующие таблицы.

## Денормализация

Материализованное представление `mv_monthly_sales` хранит заранее посчитанные месячные продажи по товарам и категориям. Это оправдано для повторяющихся отчётов, где допустим слегка устаревший снимок данных. Цена подхода — дополнительное место и необходимость обновлять представление через `REFRESH MATERIALIZED VIEW`.

Избыточное поле `order_items.product_name_snapshot` хранит название товара рядом со строкой заказа. Оно ускоряет чтение списков позиций и может сохранять исторический снимок названия. Цена подхода — риск аномалии обновления; если название должно всегда быть актуальным, нужен триггер синхронизации.

Поле `jsonb` для атрибутов товара удобно для разреженных и меняющихся характеристик, например `screen`, `ram`, `wifi`. Отдельная реляционная таблица лучше подходит, когда атрибуты требуют строгих ограничений, типизации, частых соединений или аналитики.

## OLTP vs OLAP

| Характеристика | OLTP (PostgreSQL) | OLAP (ClickHouse) |
|---|---|---|
| Модель хранения | Строковая | Колоночная |
| Типичный запрос | Один заказ, поиск клиента, короткая транзакция | Большое сканирование, агрегация, отчёт по периоду |
| Нормализация | Обычно нормализованная схема для снижения аномалий | Часто денормализованная схема для уменьшения числа JOIN |
| Транзакции | Полноценные ACID-транзакции | Оптимизация под вставку и аналитическое чтение |
| Вставка | Много небольших транзакционных вставок | Предпочтительны пакетные вставки |
| Обновление/удаление | Частые и консистентные операции | Возможны, но не являются основной нагрузкой |
| Масштабирование | Вертикальное масштабирование, реплики, партиционирование | Горизонтальное масштабирование аналитических запросов |
| Типичный сценарий использования | Оформление заказа, статус заказа, профиль клиента | Дашборды продаж, воронки, долгосрочные метрики |

## Как воспроизвести проверки

Результаты проверок сохраняются в формате текстовых файлов:

```sh
docker compose exec postgres psql -U idz1 -d idz1 -f /sql/04_oltp_queries.sql > checks/explain_before_idx.txt
docker compose exec postgres psql -U idz1 -d idz1 -f /sql/05_indexes.sql > checks/explain_after_idx.txt
docker compose exec postgres psql -U idz1 -d idz1 -f /sql/06_denorm_mv.sql > checks/mv_vs_join.txt
```

Файлы `checks/anomalies.txt` и `checks/trgm_demo.txt` содержат краткие пояснения к аномалиям и демонстрации `pg_trgm`.
