## Что сделано

В каталоге собран воспроизводимый стенд на Docker Compose для шардирования ClickHouse:

- `cluster_2x2`: 2 шарда, по 2 реплики в каждом шарде.
- 3 узла ClickHouse Keeper для кворума.
- Локальная таблица `default.events_local` на `ReplicatedMergeTree`.
- Распределенная таблица `default.events_distributed` с ключом шардирования `xxHash64(user_id)`.
- Справочная таблица `default.user_dict_local` и распределенная таблица `default.user_dict_distributed`.
- Генератор данных минимум на 2 000 000 строк clickstream-событий.
- Текстовые результаты проверок в `checks/`.
- Диаграмма топологии PlantUML в `topology.puml`.
- Дополнительная конфигурация третьего шарда и заготовки SQL/скриптов для ребалансировки.

## Почему выбран `xxHash64(user_id)`

Для clickstream-аналитики естественно группировать множество событий по одному пользователю. Ключ шардирования на основе `user_id` гарантирует, что все события конкретного пользователя окажутся на одном шарде. Это удобно для пользовательской и сессионной аналитики: `GROUP BY user_id`, соединения с пользовательскими атрибутами и выборки по одному пользователю можно выполнять локально на нужном шарде без объединения частичных состояний по одному и тому же пользователю со всех шардов.

`event_date` плохо подходит для этой нагрузки, потому что трафик сильно коррелирует со временем: новые данные в основном попадали бы в один активный день или месяц, создавая горячие шарды. Кроме того, история одного пользователя размазывалась бы по временным окнам, хотя аналитический запрос часто хочет видеть весь путь пользователя целиком.

`rand()` распределяет вставки, но не является детерминированным для одного и того же логического ключа. События одного пользователя разъехались бы по разным шардам, а агрегаты и join-операции по пользователю стали бы требовать больше межшардового обмена и хуже объяснялись бы с точки зрения предсказуемости.

`xxHash64(user_id)` детерминирован, быстр и дает почти равномерное распределение для большого множества пользователей.

## Замечания по Distributed

Глобальный `COUNT` через `events_distributed` равен сумме строк в локальных таблицах на шардах. Поскольку используется `internal_replication=true`, ClickHouse пишет в одну реплику каждого шарда, а затем данные синхронизируются между репликами внутри шарда.

`GROUP BY user_id` совпадает с ключом шардирования. Все строки одного пользователя лежат на одном шарде, поэтому каждый шард может локально построить итоговые группы по своим пользователям, а координатору остается только объединить результаты разных пользователей с разных шардов.

`GROUP BY page_url` не совпадает с ключом шардирования. Одна и та же страница может встречаться на всех шардах, поэтому каждый шард отдает частичные агрегаты координатору, а координатор затем сливает строки с одинаковым `page_url`. Это и есть shuffle/merge-фаза: данные приходится перемещать по ключу группировки, потому что физически они разложены по другому ключу.

Для join-операций справочная таблица может превратиться в проблему broadcast JOIN. Если распределенная факт-таблица соединяется с небольшой словарной таблицей, правую сторону иногда приходится рассылать на удаленные шарды. Практические варианты решения:

- Держать локальную реплицируемую справочную таблицу на каждом шарде и выполнять соединение удаленных локальных таблиц с локальной копией.
- Использовать `GLOBAL IN` или `GLOBAL JOIN`, когда правая сторона достаточно мала: инициирующий сервер один раз материализует результат и рассылает его на шарды.
- Использовать словари ClickHouse для небольших медленно меняющихся измерений.
- Шардировать измерение по `user_id` так же, как и факт-таблицу, и аккуратно использовать distributed join.
- Денормализовать стабильные атрибуты прямо в поток событий, если задержка запросов важнее гибкости обновления измерений.

## Добавление третьего шарда

`config/clickhouse/cluster.xml` содержит и `cluster_2x2`, и `cluster_3x2`. В `docker-compose.yml` уже описаны опциональные узлы `ch-s3-r1` и `ch-s3-r2`. Их можно запустить так:

```sh
docker compose --profile shard3 up -d ch-s3-r1 ch-s3-r2
```

После изменения состава кластера нужно перезапустить контейнеры ClickHouse и пересоздать `events_distributed`, чтобы она ссылалась уже на топологию из 3 шардов.

Новые вставки маршрутизируются с учетом нового числа шардов. Старые строки автоматически не переносятся: ClickHouse не выполняет прозрачную ребалансировку частей `MergeTree` при изменении топологии кластера. Пока данные явно не перемещены, старые строки остаются на шардах 1 и 2, а новые начинают распределяться по шардам 1, 2 и 3.

Подход к ребалансировке:

1. Создать локальные реплицируемые таблицы на новом шарде через `ON CLUSTER`.
2. Заморозить или приостановить запись, либо временно писать в буферную таблицу с последующим догоном.
3. Выбрать со старых шардов строки, которые по новому правилу должны попадать в шард 3, например `cityHash64(user_id) % 3 = 2`.
4. Вставить эти строки в `events_distributed` или напрямую в локальную таблицу нового шарда.
5. Удалить перенесенные строки со старых шардов через мутации, дождаться завершения и сверить количества строк и хэши.
6. Возобновить запись и сохранить текстовый лог по `system.mutations`, `system.parts` и количествам строк.

В sql/05_reshard_plan.sql лежит SQL-заготовка этого процесса.

## Файлы

- `docker-compose.yml`: 4 узла ClickHouse, 3 узла Keeper и опциональные узлы третьего шарда.
- `config/keeper/keeper*.xml`: конфигурация кворума Keeper.
- `config/clickhouse/cluster.xml`: конфигурация кластеров 2x2 и опционально 3x2.

- `sql/01_create_local.sql`: DDL локальной реплицируемой таблицы.
- `sql/02_create_distributed.sql`: DDL распределенной таблицы.

- `sql/04_queries.sql`: обязательные распределенные запросы.

- `scripts/run_checks.sh`: запись текстовых логов проверок.
- `checks/*.txt`: текстовые результаты и протоколы.
