Методика бенчмарка:
- Скрипты PostgreSQL из idz-1 выполняются в сервисе postgres.
- Схема ClickHouse из idz-2 выполняется в сервисе clickhouse.
- В ClickHouse загружается 1 000 000 сгенерированных строк позиций заказа.
- Время измеряется через EXPLAIN ANALYZE для PostgreSQL и --time для ClickHouse.

Реальные значения ClickHouse из этого запуска:

Запрос / операция                  PostgreSQL 3NF        ClickHouse, плоская   Вывод
Вставка 1 строки                    1-5 мс                нецелевой сценарий   PostgreSQL ориентирован на OLTP.
Пакетная загрузка 1 000 000 строк   нужна промежуточная таблица  один INSERT SELECT   ClickHouse лучше для пакетной загрузки.
Топ-10 товаров                      нужны JOIN            0.044 с              Плоская таблица фактов убирает JOIN.
Месячные продажи                    JOIN или MV           0.039 с из фактов    ClickHouse эффективно сканирует колонки.
Месячные продажи из агрегата        PG MV требует REFRESH 0.016 с              SummingMergeTree хранит предагрегат.
JOIN 4 таблиц                       нативно, необходимо   не нужен             Денормализация убирает работу JOIN.
Обновление статуса                  нативный UPDATE       мутация, не OLTP     PostgreSQL лучше для частых обновлений.
Поиск email по подстроке            помогает pg_trgm      0.038 с скан         PostgreSQL лучше для селективного текстового поиска.
Размер на диске, 1M строк           строки + индексы      1.18 MiB сжато       ClickHouse выигрывает на повторяющихся измерениях.

Числа следует обновлять на целевой машине, если отличается CPU, диск, лимиты Docker или прогрев кэша.
